# Design: add-interoperability-conformance

## Test Categories

### 1. Header Parsing Tests

Test vectors exercise every field of the 24-byte frame header (defined in `add-shared-frame-format`).

| Vector | Description | Expected result |
|--------|-------------|-----------------|
| `hdr_valid_v1` | Canonical v1 header | Parse succeeds |
| `hdr_bad_magic` | Magic = `0x00000000` | `ERR_CORRUPT` / rejection |
| `hdr_bad_magic_lz4` | Magic = LZ4 magic `0x04224D18` | `ERR_CORRUPT` |
| `hdr_version_0` | version byte = `0x00` | `ERR_VERSION_UNSUPPORTED` |
| `hdr_version_2` | version byte = `0x02` | `ERR_VERSION_UNSUPPORTED` |
| `hdr_unknown_flags` | flags bit 3 set | Rejection |
| `hdr_unknown_algo` | algo_id = `0xFF` | Rejection or `ERR_UNKNOWN_ALGO` |
| `hdr_truncated_at_8` | File ends at byte 8 | `ERR_TRUNCATED` |
| `hdr_truncated_at_23` | File ends at byte 23 | `ERR_TRUNCATED` |

### 2. Truncation Tests

Each valid compressed test file is truncated at multiple offsets and verified to produce `ERR_TRUNCATED` (not crash, not corrupt output, not hang).

Truncation points for each file:
- Byte 4 (mid-magic)
- Byte 12 (mid-header)
- Byte 24 (end of header, no payload)
- Byte `header_len + 1` (first payload byte)
- Byte `total_len / 2` (mid-payload)
- Byte `total_len - 1` (missing last trailer byte)

### 3. Corruption Tests

A single byte is flipped (XOR 0xFF) at specific offsets in a valid file:

| Offset | What is corrupted |
|--------|------------------|
| 2 | Magic byte 2 |
| 5 | version field |
| 6 | algo_id field |
| 16 | checksum byte 0 |
| 24 | First payload byte |
| `total_len - 4` | Trailer CRC byte 0 |

Expected result for all: either `ERR_CORRUPT` or `ERR_VERSION_UNSUPPORTED` (never silent success).

### 4. Concatenation Tests

Two valid CompressKit frames written back-to-back form a "concatenated stream". A compliant decoder MUST decode both frames and concatenate their outputs.

```
[Frame A][Frame B] → decoded_A ++ decoded_B
```

Test cases:
- Same algorithm, same content
- Same algorithm, different content
- Different algorithms (if multi-algo concatenation is supported; mark as optional)

### 5. Corpus Matrix

A fixed, reproducible set of test files covering different entropy classes:

| File | Size | Category | Generation method |
|------|------|----------|------------------|
| `corpus/empty.bin` | 0 B | Edge case | Empty |
| `corpus/single_byte.bin` | 1 B | Edge case | `\x00` |
| `corpus/all_zeros_1k.bin` | 1 KiB | Low entropy | Zeros |
| `corpus/all_zeros_64k.bin` | 64 KiB | Low entropy | Zeros |
| `corpus/repetitive_1k.bin` | 1 KiB | Low entropy | `\xAB` * 1024 |
| `corpus/rand_1k.bin` | 1 KiB | High entropy | PRNG seed=42 |
| `corpus/rand_10k.bin` | 10 KiB | High entropy | PRNG seed=42 |
| `corpus/rand_100k.bin` | 100 KiB | High entropy | PRNG seed=42 |
| `corpus/text_canterbury_alice.txt` | ~148 KiB | Natural language | Canterbury corpus |
| `corpus/binary_elf_sample.bin` | ~50 KiB | Mixed | Committed binary (see note) |

All files except `binary_elf_sample.bin` MUST be generated by `tests/gen_corpus.py --seed 42` and checked into `tests/corpus/`.

> **Note — `binary_elf_sample.bin`**: This file is a committed binary sample (a stripped copy of a small ELF utility from the host system, e.g. `/usr/bin/true`). It is checked in verbatim so CI behaviour is fully deterministic without depending on the host environment. It MUST NOT be regenerated by `gen_corpus.py`.

### 6. Cross-Language Decode Matrix

A 3×3 matrix per algorithm, where rows = encoder language and columns = decoder language:

```
         | C++ dec | Go dec | Rust dec
---------|---------|--------|----------
C++ enc  |    ✓    |   ✓    |    ✓
Go enc   |    ✓    |   ✓    |    ✓
Rust enc |    ✓    |   ✓    |    ✓
```

Applies to: Huffman, Arithmetic, and RLE for all 10 corpus files; Range for the 9 corpus files at or below 100 KiB (`text_canterbury_alice.txt` is excluded per the known Range cap).

This matrix is verified for each applicable corpus file, giving 351 verified combinations minimum: 3 algorithms × 9 pairs × 10 files = 270, plus Range × 9 pairs × 9 eligible files = 81.

## Test Infrastructure

### Test runner structure

```
tests/
├── corpus/              # Fixed corpus files (committed)
├── vectors/             # Header/corruption test vectors (JSON + binary)
│   ├── header_vectors.json
│   └── *.bin
├── conformance/
│   ├── test_header_parsing.py
│   ├── test_truncation.py
│   ├── test_corruption.py
│   ├── test_concatenation.py
│   └── test_decode_matrix.py
└── gen_corpus.py        # Corpus generator (deterministic, seed-based)
```

### Matrix result reporting

The decode matrix test SHALL output a JSON report:

```json
{
  "algorithm": "huffman",
  "corpus_file": "rand_10k.bin",
  "matrix": {
    "cpp_enc_go_dec": "PASS",
    "cpp_enc_rust_dec": "PASS",
    "go_enc_cpp_dec": "PASS",
    ...
  }
}
```

This report is for conformance verification only. `add-benchmark-governance` reuses the corpus files from `tests/corpus/` directly; it does not consume the decode matrix report.
